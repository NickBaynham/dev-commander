# Dev Commander v0.4 — Containerized Publish and Self-Hosted Deploy Design

Status: approved for planning (2026-07-23)

## Goal

Extend Dev Commander past a tagged release into shipping running software:
two skills that build and publish a container image and deploy it to a
self-hosted host, closing the lifecycle from idea to production while
keeping the skill-pack architecture and staying stack-agnostic.

## Scope

In scope for v0.4:

- **dc-publish** — build a container image for the project and push it to a
  registry (GHCR) on release. Generates a Dockerfile if the project lacks
  one.
- **dc-deploy** — deploy the published image to a self-hosted Linux host
  over SSH with docker compose.
- A generated `release.yml` GitHub Actions workflow that automates
  build → push → deploy on a version tag.

Explicitly out of scope for v0.4 (deferred to later increments):

- Registries other than GHCR (Docker Hub, ECR, GAR). GHCR pairs with the
  v0.3 GitHub Actions choice; others become a registry option later.
- Deploy targets other than a self-hosted host over SSH: container PaaS
  (Fly.io, Render), Kubernetes, serverless. These become a deploy-target
  family later, and most consume the same published image.
- Language-native package publishing (PyPI, npm). A different audience
  (library authors) from "ship a running app"; a later increment.
- Rollback orchestration, blue/green or canary deploys, and multi-host
  fan-out. v0.4 deploys one image to one host.

## Decisions

These extend the existing DC1-DC14 series in `planning/plan.md`.

| # | Decision |
| --- | --- |
| DC15 | dc-publish and dc-deploy are Markdown-only, extending DC11. Their templates ship in the plugin; the agent runs docker, git, and ssh by following each SKILL.md. |
| DC16 | The container image is the deployment artifact. GHCR (`ghcr.io/<owner>/<project>`) is the default registry in v0.4, matching the GitHub-centric CI choice (DC12). Other registries are deferred. |
| DC17 | The deploy target is a self-hosted Linux host running docker compose over SSH; vendor-neutral. PaaS and Kubernetes targets are deferred to a later target family. |
| DC18 | Neither skill embeds secrets. Registry auth and SSH credentials come from the user's environment locally, or GitHub Actions secrets in CI; the skills reference them, never store them. |

## Architecture

Two Markdown-only skills unified by one artifact — the container image — so
publish and deploy describe the same thing. Only the Dockerfile is
stack-specific; the production compose file and the release workflow are
stack-agnostic, because the Dockerfile encapsulates the stack.

Stack detection reuses the existing rule (`pyproject.toml` present means
python, `package.json` means node-ts, `go.mod` means go; ask if none or
more than one).

### Component: dc-publish skill

Command: `/dc:publish`.

Behavior:

1. Detect the stack.
2. Ensure a Dockerfile exists. If none, generate one from
   `templates/docker/<stack>/Dockerfile.tmpl`, substituting the project
   name. Never overwrite an existing Dockerfile.
3. Build the image and tag it `ghcr.io/<owner>/<project>:<version>` and
   `:latest`, where `<version>` is the project's current version (from the
   manifests) and `<owner>/<project>` derive from the git remote and
   project name.
4. Push both tags to GHCR. Publishing assumes a clean, released tree; it
   pairs with dc-release (you publish the version you tagged).
5. Registry authentication comes from the environment (`docker login`
   already done, or `GITHUB_TOKEN` in CI). dc-publish never stores or
   prints a credential (DC18).
6. Write a publish record to `.dev-commander/deployments/NNNN-<slug>.md`:
   the image reference and tags pushed, and the outcome.
7. dc-publish is the single source of truth for the image reference and
   the build/push commands. It supplies the build-and-push job the release
   workflow embeds.

### Component: dc-deploy skill

Command: `/dc:deploy`.

Behavior:

1. Ensure `docker-compose.prod.yml` exists. If none, generate it from
   `templates/deploy/docker-compose.prod.yml.tmpl`, substituting the
   project name and the image reference dc-publish documents. Never
   overwrite an existing file.
2. Establish the target: the host and SSH user come from a Deployment
   section in the workspace `project.md` (a committed, reviewable location);
   the image comes from GHCR. If the host is not configured there, ask the
   user and offer to record it in `project.md`; never guess.
3. Deploy: over SSH, pull the image and run
   `docker compose -f docker-compose.prod.yml up -d`.
4. SSH credentials and registry auth come from the environment (local) or
   GitHub Actions secrets (CI). dc-deploy references them, never embeds
   them (DC18).
5. Write a deploy record to `.dev-commander/deployments/NNNN-<slug>.md`:
   the image deployed and the host it went to, and the outcome.
6. dc-deploy supplies the deploy job the release workflow embeds.

### Component: templates

Five bundled templates, the whole surface:

- `templates/docker/python/Dockerfile.tmpl`
- `templates/docker/node-ts/Dockerfile.tmpl`
- `templates/docker/go/Dockerfile.tmpl` (stack-specific base images and
  build steps; each produces a runnable image).
- `templates/deploy/docker-compose.prod.yml.tmpl` (stack-agnostic; runs the
  published image, substitutes `{{project_name}}` and the image reference).
- `templates/deploy/release.yml.tmpl` (stack-agnostic GitHub Actions
  workflow, triggered on a version tag: build the image, push to GHCR, then
  deploy over SSH).

Only `{{project_name}}` and the image reference are substituted; no
`{{project_description}}` is needed in these artifacts.

### Component: workspace directory

A new `.dev-commander/deployments/` directory takes the workspace from nine
to ten directories, holding publish and deploy records as `NNNN-<slug>.md`.
This is the same coordinated change design/, learning/, and security/ used:

- The workspace template gains `deployments/.gitkeep`.
- `status.py` DIRS, `test_dc_core.py` DIRS, and
  `test_lifecycle_integration.py` WORKSPACE_DIRS all gain `deployments`,
  placed last (after `handoff`), since deploy is the terminal step.
- `init_workspace.py` creates it automatically (recursive template copy; no
  code change beyond the template file).

### next_step integration

The lifecycle recommender gains one workspace-keyed state: when the cycle is
otherwise complete (lessons captured and a security scan present) but
`deployments/` holds no record, recommend the ship sequence — `/dc:release`,
then `/dc:publish` and `/dc:deploy`. Once a deployment record exists, the
cycle is truly complete. The generated `release.yml` lives in `.github/`,
outside the workspace next_step reads, so like `/dc:ci` it is mentioned as
advice, not a separate gated state.

## Data flow

```
/dc:publish ── detect stack ──► Dockerfile (generate if absent)
                                     │ docker build
                                     ▼
                        ghcr.io/<owner>/<project>:<version>  (pushed image)
                                     │
                                     ▼
                     .dev-commander/deployments/NNNN-<slug>.md  (record)

/dc:deploy  ── image ref ──► docker-compose.prod.yml (generate if absent)
                                     │ ssh host: pull + compose up -d
                                     ▼
                          running container on the target host
                                     │
                                     ▼
                     .dev-commander/deployments/NNNN-<slug>.md  (record)

release.yml (on tag) ──► build ──► push to GHCR ──► ssh deploy
```

The image reference dc-publish documents is embedded verbatim in the compose
file and the release workflow, so publish, deploy, and the pipeline all name
the same artifact.

## Error handling

- Missing docker, git, or ssh: report the missing tool and stop; never crash.
- Existing Dockerfile / docker-compose.prod.yml / release workflow: generate
  only when absent; report and keep the existing file otherwise (matching
  dc-scaffold's and dc-ci's non-overwrite rule).
- Unconfigured deploy host: dc-deploy asks for the host rather than guessing.
- Missing credentials: report that registry auth or SSH access is not
  configured and stop; never embed or prompt for a secret to store.
- Missing workspace: direct the user to `/dc:init` first, matching the other
  workspace-writing skills.

## Testing

Pure-Python, following the existing fast suite; no new dependency (pyyaml is
already present from v0.3).

- `test_dc_skills.py`: add dc-publish and dc-deploy entries to EXPECTED
  (frontmatter present, name matches directory, required content markers
  such as `/dc:publish`, `ghcr`, `/dc:deploy`, `docker compose`).
- A new `test_dc_deploy.py`:
  - The three `templates/docker/<stack>/Dockerfile.tmpl`,
    `templates/deploy/docker-compose.prod.yml.tmpl`, and
    `templates/deploy/release.yml.tmpl` all exist.
  - `release.yml.tmpl` parses as valid YAML, triggers on a version tag
    (`on: push: tags`), and contains build, push, and deploy steps.
  - Image-reference cross-check: the compose file and the release workflow
    reference the same `ghcr.io/.../<project>` image-name convention
    dc-publish documents.
- `test_dc_core.py`: DIRS gains `deployments`; the init test asserts the
  directory is created.
- No integration test runs a real `docker build`, registry push, or SSH
  deploy; those need docker, a registry, and a host, and are verified by the
  manual health-check discipline (build the image, run it locally, deploy to
  a scratch host), exactly as scaffold build/run and CI are verified today.

## Anticipated task decomposition

Rough shape for the implementation plan (writing-plans will detail it):

1. `deployments/` workspace directory + status/init/DIRS + tests.
2. Dockerfile template family (three stacks) + tests.
3. dc-publish skill (Dockerfile-generate, build, push to GHCR) + dc-skills
   test entry.
4. docker-compose.prod.yml and release.yml templates + YAML and image-ref
   cross-check tests.
5. dc-deploy skill (compose-generate, SSH deploy) + dc-skills test entry.
6. next_step ship state + tests.
7. Documentation and release: README command table, docs/ updates, plan
   Decisions DC15-DC18, and a v0.4.0 release via dc-release.

## Constraints inherited from the project

All existing Global Constraints apply: pdm; Make targets; no emojis
anywhere; the plan is the source of truth (fix the plan first); Markdown
skills own their behavior; templates ship inside the plugin; the
`.dev-commander/` workspace is committed to the consuming project's git;
product-domain-agnostic vocabulary; the `dc-` skill prefix and `/dc:`
command namespace; platform support macOS, Linux, WSL2, Git Bash.
