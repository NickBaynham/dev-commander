# Dev Commander v0.3 — CI Gates and Security Scanning Design

Status: approved for planning (2026-07-22)

## Goal

Extend Dev Commander past release into continuous integration: two new
skills that generate a CI pipeline gating pull requests and run security
scans, keeping the skill-pack architecture and staying stack-agnostic.

## Scope

In scope for v0.3:

- **dc-ci** — generate a GitHub Actions workflow that runs the scaffold's
  uniform Make targets plus a security scan on push and pull request.
- **dc-secscan** — run the project's stack-native dependency scanner and a
  universal secret scan on demand, write a workspace report, and supply the
  scan step dc-ci embeds.

Explicitly out of scope for v0.3 (deferred to later increments):

- Continuous deployment to any environment (cloud, Kubernetes, servers).
- Publishing build artifacts (container images, PyPI/npm packages) — pairs
  with dc-release in a future version.
- CI providers other than GitHub Actions.
- Static application security testing (SAST): bandit, gosec, semgrep. Higher
  false-positive rate and more framework-specific than the two categories
  shipped here.

## Decisions

These extend the existing DC1–DC10 series in `planning/plan.md`.

| # | Decision |
| --- | --- |
| DC11 | dc-ci and dc-secscan are Markdown-only, extending DC3. No scanners or CI tools are vendored; the agent runs the project's stack-native tools and writes config by following each SKILL.md. |
| DC12 | GitHub Actions is the only CI provider in v0.3. Other providers are a deferred increment, structured as `templates/ci/<provider>/` if added. |
| DC13 | Security gate default: fail CI on high or critical dependency vulnerabilities and on any committed secret; report-only below high severity. SAST is deferred. |
| DC14 | dc-secscan degrades gracefully when a scanner is absent: it reports "scanner not installed, install with X", continues, and records which scanners actually ran. |

## Architecture

Two Markdown-only skills, one new GitHub Actions template family, and one new
workspace directory. Both skills stay stack-agnostic by keying off two things
the scaffold already provides:

1. The uniform Make surface every stack family ships: `install`, `lint`,
   `test`, `build`.
2. A stack-detection rule: `pyproject.toml` present → python, `package.json`
   → node-ts, `go.mod` → go.

The division of labor mirrors dc-handoff and dc-release: dc-secscan owns the
"what and how" of scanning and is usable on its own; dc-ci assembles that
into a gate and consumes dc-secscan's scan step rather than redefining it.

### Component: dc-secscan skill

Command: `/dc:scan`.

Behavior:

1. Detect the stack from project files (rule above).
2. Run the dependency scanner for the stack and the universal secret scan:
   - Dependency (SCA): python → `pip-audit`, node-ts → `npm audit`,
     go → `govulncheck`.
   - Secrets (all stacks): `gitleaks` over the repository.
3. When a scanner is not installed, report the install command and continue
   (DC14); never crash on a missing tool.
4. Write a report to `.dev-commander/security/NNNN-<slug>.md`, where NNNN is
   the next zero-padded sequence number, with: the scanners run, findings
   grouped by severity (package or file location, advisory reference), and a
   verdict — clean, or issues found.
5. Never auto-fix. A dependency bump to resolve a vulnerability becomes a
   `/dc:plan` follow-up; the report recommends the fix, the user decides.
6. dc-secscan is the single source of truth for how to scan each stack: dc-ci
   embeds the dependency-scan commands verbatim and runs the same gitleaks
   engine (the official gitleaks GitHub Action) for the secret scan.

The report's verdict and severity vocabulary follow DC13: high or critical
dependency findings and any committed secret are gating-level; lower findings
are informational.

### Component: dc-ci skill

Command: `/dc:ci`.

Behavior:

1. Detect the stack (same rule).
2. Read `templates/ci/github/<stack>/ci.yml.tmpl`, substitute
   `{{project_name}}`, and write it to `.github/workflows/ci.yml` in the
   project. Never overwrite an existing workflow; if one exists, report it
   and stop.
3. Report what was written and recommend running `/dc:scan` locally before
   relying on the gate.

Each generated workflow triggers on `push` and `pull_request` and runs, in
order: checkout, the language setup action for the stack, `make install`,
`make lint`, `make test`, `make build`, then the dc-secscan scan steps. The
security step fails the build per DC13.

### Component: CI template family

Location: `templates/ci/github/<stack>/ci.yml.tmpl`, three files
(`python`, `node-ts`, `go`), mirroring the scaffold family pattern.

Each template is a complete GitHub Actions workflow:

- `on: [push, pull_request]`.
- A single job on `ubuntu-latest`.
- Steps: `actions/checkout`, the stack's setup action
  (`actions/setup-python`, `actions/setup-node`, or `actions/setup-go`),
  `make install lint test build`, the dependency scan for the stack, and the
  `gitleaks` secret scan.
- Only `{{project_name}}` is substituted; no `{{project_description}}` is
  needed in a workflow.

The Make targets a template names must be a subset of the targets in that
stack's scaffold `Makefile.tmpl` — enforced by a test (below).

### Component: workspace directory

A new `.dev-commander/security/` directory takes the workspace from eight to
nine directories, holding scan reports as `NNNN-<slug>.md`. This is the same
coordinated change design/ and learning/ used:

- The workspace template gains `security/.gitkeep`.
- `status.py` DIRS and `test_dc_core.py` DIRS both gain `security` (placed
  before `handoff`, consistent with the lifecycle ordering).
- `init_workspace.py` creates it automatically (it copies the template tree
  recursively; no code change beyond the template file).

### next_step integration

The lifecycle recommender (`next_step.py`) gains one workspace-keyed check:
when the cycle is otherwise complete but `.dev-commander/security/` holds no
scan report, recommend `/dc:scan` before `/dc:release`. dc-ci is not gated by
next_step because a `.github/` workflow lives outside the workspace that
next_step reads exclusively; dc-ci is surfaced as advice in the relevant
recommendation text, not as its own state.

## Data flow

```
/dc:scan  ── detect stack ──► run pip-audit|npm audit|govulncheck + gitleaks
                                         │
                                         ▼
                         .dev-commander/security/NNNN-<slug>.md  (report + verdict)

/dc:ci    ── detect stack ──► templates/ci/github/<stack>/ci.yml.tmpl
                                         │ substitute {{project_name}}
                                         ▼
                              .github/workflows/ci.yml   (push/PR gate:
                              make install lint test build + scan steps)
```

dc-ci embeds the scan commands that dc-secscan documents, so "how to scan
this stack" has one source of truth.

## Error handling

- Missing scanner: reported with an install hint, scan continues (DC14).
- Existing `.github/workflows/ci.yml`: dc-ci reports and stops; it never
  overwrites (matching dc-scaffold's non-overwrite rule).
- Unknown or ambiguous stack (no `pyproject.toml`/`package.json`/`go.mod`, or
  more than one): the skill asks the user which stack rather than guessing.
- Missing workspace for the security report: dc-secscan directs the user to
  `/dc:init` first, matching the other workspace-writing skills.

## Testing

Pure-Python, following the existing fast suite; one new dev dependency.

- `test_dc_skills.py`: add dc-ci and dc-secscan entries to EXPECTED
  (frontmatter present, name matches directory, required content markers such
  as `/dc:ci`, `.github/workflows`, `/dc:scan`, `security/`).
- New `test_dc_ci.py`:
  - All three `templates/ci/github/<stack>/ci.yml.tmpl` exist.
  - Each parses as valid YAML (adds **pyyaml** as a dev dependency).
  - Each triggers on both `push` and `pull_request`.
  - Each runs the uniform Make targets and embeds the correct scanner
    commands for its stack.
  - Cross-check: every Make target named in a CI template exists in that
    stack's scaffold `Makefile.tmpl`.
- `test_dc_core.py`: DIRS gains `security`; the init test asserts the
  directory is created.
- No integration test executes a real GitHub Actions run; the contract is
  validated by YAML well-formedness plus the Make-target cross-check.

## Anticipated task decomposition

Rough shape for the implementation plan (writing-plans will detail it):

1. `security/` workspace directory + status/init/DIRS + tests.
2. dc-secscan skill (SKILL.md + dc-skills test entry).
3. CI template family (three GitHub Actions templates) + `test_dc_ci.py` +
   pyyaml dev dependency.
4. dc-ci skill (SKILL.md + dc-skills test entry).
5. next_step `/dc:scan` recommendation + tests.
6. Documentation and release: README command table, CHANGELOG, AGENTS.md
   identity prose, plan Decisions DC11–DC14, and a v0.3.0 release via
   dc-release.

## Constraints inherited from the project

All existing Global Constraints apply: pdm; Make targets; no emojis anywhere;
plan is the source of truth (fix the plan first); Markdown skills own their
behavior; templates ship inside the plugin; the `.dev-commander/` workspace is
committed to the consuming project's git; product-domain-agnostic vocabulary;
`dc-` skill prefix and `/dc:` command namespace.
