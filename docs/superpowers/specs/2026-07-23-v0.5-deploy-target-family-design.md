# Dev Commander v0.5 — Deploy-Target Family Design

Status: approved for planning (2026-07-23)

## Goal

Turn dc-deploy from a single-target skill (self-hosted docker compose over
SSH) into a target family, and add Fly.io as the second target — the "no VM
to manage" path that complements the existing "I have a host" path — keeping
the container image the single deployment artifact across targets.

## Scope

In scope for v0.5:

- Restructure the existing SSH deploy templates into a per-target family
  layout (`templates/deploy/ssh/`).
- Select the active target with a `target:` key in `project.md`'s Deployment
  section (asked and recorded if absent, like the host).
- Add a **Fly.io** target: a `fly.toml` config and a release workflow that
  deploys the published GHCR image with `flyctl`.

Out of scope for v0.5 (deferred):

- Kubernetes and other deploy targets (a later family member).
- Registries other than GHCR.
- Language-native package publishing.
- Any new workspace directory, next_step change, or new skill.

## Decisions

These extend the existing DC1-DC18 series in `planning/plan.md`.

| # | Decision |
| --- | --- |
| DC19 | dc-deploy is a target family. Targets live under `templates/deploy/<target>/`, and the active target is selected by a `target:` key in `project.md`'s Deployment section (asked and recorded if absent, like the host). v0.5 ships `ssh` (moved) and `fly`. |
| DC20 | The Fly.io target deploys the GHCR image dc-publish built (`flyctl deploy --image ...`), not a Fly-side build, keeping the container image the single artifact across targets. `FLY_API_TOKEN` is referenced from the environment or a GitHub Actions secret, never stored (DC18 holds). |

## Architecture

dc-deploy stays one Markdown-only skill that now branches by target. The
templates reorganize into per-target directories, exactly like the scaffold
(`templates/scaffold/<stack>/`) and CI (`templates/ci/github/<stack>/`)
families:

```
templates/deploy/
├── ssh/                          # the current target, moved here
│   ├── docker-compose.prod.yml.tmpl
│   └── release.yml.tmpl          # scp + ssh deploy
└── fly/                          # NEW
    ├── fly.toml.tmpl             # Fly app config
    └── release.yml.tmpl          # flyctl deploy
```

Target selection: dc-deploy reads `target:` (`ssh` or `fly`) from the
Deployment section of the workspace `project.md`. If absent, it asks and
offers to record it — the same ask-and-record pattern the deploy host
already uses. No new workspace directory; next_step is untouched (the ship
state already recommends `/dc:deploy`; which target it uses is dc-deploy's
internal concern).

Generalized `/dc:deploy` behavior:

1. Read the target from `project.md` (ask and record if absent).
2. Generate that target's config if absent (compose for ssh, `fly.toml` for
   fly) — never overwriting.
3. Deploy via the target's mechanism (ssh + docker compose, or `flyctl
   deploy --image ...`).
4. Generate that target's `.github/workflows/release.yml` from the target's
   template.
5. Write a deploy record to `.dev-commander/deployments/NNNN-<slug>.md`.

The image reference `ghcr.io/{{repo_owner}}/{{project_name}}` is shared
across targets: Fly deploys that exact image rather than building its own, so
the image-reference contract carries straight over.

### Component: the Fly.io target

- `templates/deploy/fly/fly.toml.tmpl` — a minimal Fly app config (app name
  `{{project_name}}`, an internal port, a health check) a fresh project can
  deploy against unedited.
- `templates/deploy/fly/release.yml.tmpl` — on a `v*` tag: dc-publish's
  build-and-push to GHCR (with the same version normalization the SSH
  workflow uses), then `flyctl deploy --image
  ghcr.io/{{repo_owner}}/{{project_name}}:<version> --app {{project_name}}`,
  authenticated with a `FLY_API_TOKEN` GitHub Actions secret.

## Migration

The existing `templates/deploy/docker-compose.prod.yml.tmpl` and
`release.yml.tmpl` move into `templates/deploy/ssh/` (a `git mv`). dc-deploy's
SKILL.md template paths and `tests/test_dc_deploy.py`'s `DEPLOY` constant
repoint to the `ssh/` subdirectory. The SSH behavior itself is unchanged.

## Error handling

- Unknown or missing `target:` in `project.md`: ask the user which target
  and offer to record it; never guess.
- An unsupported target value: report the supported set (`ssh`, `fly`) and
  stop.
- Missing `flyctl` (Fly target) or `docker`/`ssh` (SSH target): report the
  missing tool and stop; never crash.
- Existing target config or `.github/workflows/release.yml`: generate only
  if absent; report and keep the existing file otherwise.
- Missing credentials: report that the token or SSH access is not
  configured; never embed or store a secret (DC18).

## Testing

Pure-Python, following the existing suite; no new dependency (`tomllib` is
stdlib in Python 3.12 for parsing `fly.toml`; `pyyaml` is already present).

- The restructure repoints `tests/test_dc_deploy.py`'s `DEPLOY` constant to
  `templates/deploy/ssh/`; the existing raw-YAML, image-ref cross-check, and
  placeholder-guard tests carry over unchanged.
- New Fly tests: `fly.toml.tmpl` exists and parses as valid TOML with an
  `app` key; the Fly `release.yml.tmpl` parses as YAML, triggers on a `v*`
  tag, references the shared image string, and invokes `flyctl deploy`; a
  placeholder guard (no `{{project_name}}`/`{{repo_owner}}` survive
  rendering; GitHub `${{ }}` expressions do).
- A structural test asserting `templates/deploy/` contains exactly the target
  directories `ssh` and `fly`, the same guard the scaffold family uses.
- No executed `fly deploy` (it needs a Fly account and token) — correctness
  is validated by template well-formedness and the image-ref cross-check,
  exactly as the SSH release path and the CI workflow are.

## Anticipated task decomposition

1. Restructure the SSH templates into `templates/deploy/ssh/`; repoint the
   test `DEPLOY` constant and dc-deploy's SKILL paths.
2. Generalize dc-deploy's SKILL to read `target:` from `project.md` and
   branch by target.
3. The Fly `fly.toml.tmpl` and `release.yml.tmpl` templates + Fly and
   family-structure tests.
4. Documentation (README, docs/commands.md, docs/lifecycle.md note the target
   choice) and the v0.5.0 release via dc-release.

## Constraints inherited from the project

All existing Global Constraints apply: pdm; Make targets; no emojis
anywhere; the plan is the source of truth; Markdown skills own their
behavior; templates ship inside the plugin; product-domain-agnostic
vocabulary; the `dc-` prefix and `/dc:` namespace; secrets referenced, never
stored (DC18).
